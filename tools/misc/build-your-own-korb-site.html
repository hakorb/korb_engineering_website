Build Your Own Website

Hi. I'm Harrison. This little site started as a hobby and grew to 230+ free tools I built for myself, my family, and my engineering work. Everything is hosted for free and runs in your browser.

This guide will walk you through how to build your own version of it, step by step, even if you've never written a line of code. Take your time, tick off each step, and come back whenever you need to.

01

Overview & Philosophy

5 min read

korb.engineering is a single website made of many small, self-contained HTML tools. Each tool is one file. Each file has its own HTML, CSS, and JavaScript. No server. No database. No build process. You open the file in your browser, and it works.

The philosophy is simple: small things that last. A tool you wrote in 2025 should still open and run in 2045, without maintenance, without updates, without anything breaking because some npm package got renamed.

The Korb Site Formula

  • One repository. Everything lives in a single GitHub folder.
  • One shell page. A single index.html that loads the right tool based on the URL.
  • Many self-contained tools. Each tool is a standalone HTML file in tools/category/tool-name.html.
  • Iframes for isolation. Tools load inside an iframe so one broken tool can't take down the whole site.
  • Free hosting. GitHub Pages serves it for $0 forever.
Let's Be Honest About Tradeoffs
This approach is simple and durable, but it's not perfect. There is no backend, so you can't store user accounts or secrets. SEO isn't great for in-iframe tools. Files can get big because each one repeats its own CSS. And you'll bump into browser limits if you try to do heavy data processing. For personal tools and small projects, though, it's wonderful. If you outgrow it, you'll know.

Ready? Let's build something.

What You Need

2 min

Good news: almost everything on this list is free. You probably already own most of it.

  • A computer. Mac, Windows, or Linux. Any machine from the last ten years is plenty.
  • A web browser. Chrome, Firefox, Safari, or Edge. You've already got one.
  • A free GitHub account. GitHub is where your code lives and where your site gets hosted. Signup is free at github.com.
  • A text editor. I recommend VS Code. It's free, fast, and the industry standard. If you want to start even simpler, you can edit files directly on github.com with zero installs.
  • (Optional) A custom domain. $10–15/year at Namecheap or Porkbun. You don't need one to start. A free username.github.io URL works fine.
  • (Optional) Claude Code. Anthropic's AI coding assistant. If you're not a developer, this is a game-changer. We'll cover it in Step 9.
Pro Tip
You can complete Steps 1–7 without installing anything at all. GitHub lets you create, edit, and save files directly on the website. If that feels more comfortable, start there and install VS Code later.

Step 1: Create a GitHub Account

5 min

GitHub is basically Dropbox for code. It stores your files, keeps a history of every change you make, and (most importantly) can serve your site to the whole internet for free.

What To Do

  1. Go to github.com/signup.
  2. Enter your email and create a password.
  3. Choose your username carefully. This will become part of your website URL, like yourname.github.io. Short and memorable is best. You can change it later but it's a hassle.
  4. Verify your email address when GitHub sends you the verification code.
  5. On the plan page, pick the Free plan. That's all you need.
About Your Username
If your username is janedoe, your free site URL will be janedoe.github.io. Pick something you'd be happy seeing in a browser address bar.

Step 2: Create Your Repository

3 min

A repository (or "repo") is just a folder for one project. We'll make one that GitHub Pages will automatically turn into a website.

What To Do

  1. While logged in, click the + in the top right of github.com and pick New repository.
  2. For the Repository name, type yourusername.github.io — replacing yourusername with your actual username. The name must match exactly.
  3. Set it to Public. (Private repos can't use free GitHub Pages.)
  4. Check the box to Add a README file. This gives you something to start with.
  5. Click Create repository.
Why the Funny Name?
When a repo is named yourusername.github.io, GitHub treats it as a special "user site" and hosts it at https://yourusername.github.io automatically. It's a convention that just works.
Alternate Path
If you plan to use a custom domain (like mycoolsite.com), you can name the repo anything you want. We'll cover custom domains in Step 10.

Step 4: Your First index.html

10 min

Now the fun starts. We'll create a single HTML file that says "Hello, world" in the same VFD dark terminal theme as this site. It's short, clean, and you can copy-paste it.

Create the File

  1. Go to your repo on github.com (the one named yourusername.github.io).
  2. Click Add file → Create new file.
  3. Name it index.html exactly.
  4. Paste the code below into the editor box.
  5. Scroll down, add a commit message like "first page", and click Commit changes.
index.html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0, viewport-fit=cover">
  <title>My Site</title>
  <link href="https://fonts.googleapis.com/css2?family=JetBrains+Mono:wght@400;600&display=swap" rel="stylesheet">
  <style>
    :root{
      --bg:#050a0f; --surface:#0a1018; --border:#1a2535;
      --cyan:#00d4ff; --text:#e2e8f0;
    }
    *{box-sizing:border-box;margin:0;padding:0}
    body{
      background:var(--bg); color:var(--text);
      font-family:'JetBrains Mono',monospace;
      min-height:100vh;
      display:flex; align-items:center; justify-content:center;
      padding:1rem;
    }
    .card{
      background:var(--surface);
      border:1px solid var(--border);
      border-radius:8px;
      padding:2rem;
      max-width:500px; width:100%;
      text-align:center;
    }
    h1{
      color:var(--cyan);
      font-size:1.4rem;
      letter-spacing:2px;
      text-transform:uppercase;
      margin-bottom:0.75rem;
    }
    p{ font-size:0.9rem; line-height:1.6; color:#94a3b8 }
  </style>
</head>
<body>
  <div class="card">
    <h1>Hello, World</h1>
    <p>This is my new site. It works.</p>
  </div>
</body>
</html>

What Each Part Does

  • <!DOCTYPE html> — tells the browser "this is a modern HTML file".
  • <head> — invisible setup: the page title, the font, the stylesheet.
  • <meta viewport> — the magic line that makes the page work on phones. Never skip it.
  • <style> — all of your CSS. Self-contained.
  • <body> — what people actually see on the page.

Step 5: Enable GitHub Pages

3 min

Now we flip the switch that turns your repo into a real, live website.

What To Do

  1. In your repo, click Settings (top right).
  2. In the left sidebar, scroll down and click Pages.
  3. Under "Source", pick Deploy from a branch.
  4. Under "Branch", pick main and the folder / (root). Click Save.
  5. Wait about a minute. GitHub needs a moment to build and deploy.
  6. Open a new tab and go to https://yourusername.github.io.
Celebrate
That's it. You made a website. A real one, on the real internet, at a real URL you can text to your mom. This is the biggest hurdle in the whole tutorial and you just cleared it.
Not Loading?
Give it another minute or two. Make sure the file is named index.html exactly (lowercase, with the "l"). Refresh with Cmd/Ctrl+Shift+R to bust the cache. Check back in the Pages settings — if it worked, you'll see a green message with your URL.

Step 6: The SPA Architecture

10 min read

OK, you have a page. Now let's talk about the structure that makes korb.engineering scale to hundreds of tools without getting messy.

SPA stands for Single Page Application. Sounds fancy; it's not. It just means one main HTML page that swaps its contents based on the URL — instead of every click loading a whole new page from the server.

Harrison's Pattern, In Plain English

  • One index.html — the "shell". Contains the header, the navigation, and an empty container.
  • One app.js — the tiny JavaScript file that decides what to show in the container.
  • A list of all your tools in a JavaScript object called SECTIONS.
  • Each tool is its own self-contained HTML file at tools/category/my-tool.html.
  • When you click a tool, the shell puts an <iframe> in the container and points it at that tool's file.

Why Iframes?

Iframes are like little sandboxes. Each tool's CSS stays inside its own iframe, so a color you pick in one tool can't mess up another tool. If one tool crashes, the shell around it keeps working. And because each tool is self-contained, you can open any file directly in your browser, no shell needed — super handy for testing.

Hash Routing In ~15 Lines

The URL's hash is everything after the #. Changing it doesn't reload the page. So we use it as our router:

app.js
const SECTIONS = {
  home:     { title: 'Home',      tool: null },
  calc:     { title: 'Calculator', tool: 'tools/calc.html' },
  todo:     { title: 'Todo List',  tool: 'tools/todo.html' },
};

function render() {
  const id = (location.hash || '#home').slice(1);
  const section = SECTIONS[id] || SECTIONS.home;
  const container = document.getElementById('view');
  container.innerHTML = section.tool
    ? `<iframe src="${section.tool}" style="width:100%;height:100vh;border:0"></iframe>`
    : `<h1>${section.title}</h1>`;
}

window.addEventListener('hashchange', render);
window.addEventListener('load', render);

That's the whole router. Add a tool? Add a line to SECTIONS. Link to it? Use <a href="#calc">Calc</a>. That's the entire architecture.

Step 7: Your First Tool

15 min

Let's build a tiny tip calculator. It's one file. It has inputs, it does math, and it shows you the answer. A perfect first tool.

Create tools/tip-calc.html

In your repo, click Add file → Create new file. For the name, type tools/tip-calc.html — the slash tells GitHub to put it in a folder called tools. Paste this code:

tools/tip-calc.html
<!DOCTYPE html>
<html lang="en">
<head>
<meta charset="UTF-8">
<meta name="viewport" content="width=device-width,initial-scale=1,viewport-fit=cover">
<title>Tip Calculator</title>
<link href="https://fonts.googleapis.com/css2?family=JetBrains+Mono:wght@400;600&display=swap" rel="stylesheet">
<style>
  :root{
    --bg:#050a0f; --surface:#0a1018; --border:#1a2535;
    --cyan:#00d4ff; --text:#e2e8f0; --dim:#94a3b8;
  }
  *{box-sizing:border-box;margin:0;padding:0}
  body{
    background:var(--bg); color:var(--text);
    font-family:'JetBrains Mono',monospace;
    padding:1.25rem; min-height:100vh;
  }
  h1{
    color:var(--cyan); font-size:1rem;
    letter-spacing:2px; text-transform:uppercase;
    margin-bottom:1rem;
  }
  .card{
    background:var(--surface); border:1px solid var(--border);
    border-radius:8px; padding:1.25rem;
    max-width:400px;
  }
  label{
    display:block; font-size:0.72rem;
    color:var(--dim); text-transform:uppercase;
    letter-spacing:1px; margin-bottom:0.4rem;
  }
  input{
    width:100%; font-family:inherit; font-size:16px;
    background:#02060a; border:1px solid var(--border);
    color:var(--text); padding:0.75rem;
    border-radius:4px; min-height:44px;
    margin-bottom:1rem;
  }
  input:focus{ outline:none; border-color:var(--cyan) }
  .result{
    border-top:1px solid var(--border);
    padding-top:1rem; font-size:1.25rem;
    color:var(--cyan); font-weight:600;
  }
  .result b{ font-size:1.4rem }
</style>
</head>
<body>
  <h1>Tip Calculator</h1>
  <div class="card">
    <label>Bill Amount ($)</label>
    <input id="bill" type="number" value="20" step="0.01">
    <label>Tip %</label>
    <input id="tip" type="number" value="18" step="1">
    <div class="result">
      Tip: $<b id="tipOut">0.00</b><br>
      Total: $<b id="totalOut">0.00</b>
    </div>
  </div>
<script>
  const bill = document.getElementById('bill');
  const tip = document.getElementById('tip');
  const tipOut = document.getElementById('tipOut');
  const totalOut = document.getElementById('totalOut');
  function calc(){
    const b = parseFloat(bill.value) || 0;
    const t = parseFloat(tip.value) || 0;
    const tipAmt = b * (t / 100);
    tipOut.textContent = tipAmt.toFixed(2);
    totalOut.textContent = (b + tipAmt).toFixed(2);
  }
  bill.addEventListener('input', calc);
  tip.addEventListener('input', calc);
  calc();
</script>
</body>
</html>

Commit the file. Wait a minute. Visit https://yourusername.github.io/tools/tip-calc.html. You just built your first tool.

Register It In The SPA

If you've set up an app.js from Step 6, add your tool to the SECTIONS list:

app.js
const SECTIONS = {
  home: { title: 'Home', tool: null },
  tip:  { title: 'Tip Calculator', tool: 'tools/tip-calc.html' },
};

Now a link to #tip will load your calculator inside the shell. Simple as that.

Touch target44px minimum, every button.
Line height1.45–1.6 for readable prose.
Border radius4–8px. Nothing round.
Letter spacing+1px on UPPERCASE labels.
Transitions.15–.25s ease. Short.
Mobile firstBuild narrow, add breakpoints up.
The Rule Of One Font
Using only JetBrains Mono keeps everything feeling consistent and makes every tool look like part of the same family. One font, ten colors, infinite possibilities.

Step 9: Using AI to Build Faster

10 min

Here's the honest truth: I didn't hand-write all 230+ tools. I'd still be on number 20. I use Claude Code, Anthropic's command-line AI coding assistant, and it's the single biggest accelerator in my toolbox.

What It Is

Claude Code is an AI that can read your files, write new ones, and follow your instructions in plain English. You tell it "build a tool that does X", and it writes the whole HTML file for you. Then you review it and tweak it. Get it at claude.com/claude-code.

How To Prompt It

The trick is being specific. Vague prompts give vague code. Here's a template I use constantly:

Prompt Template
Build a self-contained HTML tool at tools/category/tool-name.html
that does [describe the thing clearly in 2-3 sentences].

Requirements:
- Self-contained: all HTML, CSS, and JS in one file
- Vanilla JS only, no external libraries
- VFD dark theme: bg #050a0f, cyan #00d4ff, JetBrains Mono font
- Mobile-first, 16px+ inputs, 44px+ touch targets
- Use localStorage to persist user data under key "mytool_v1"
- Match the style of tools/misc/tip-calc.html

Features:
1. [feature one]
2. [feature two]
3. [feature three]

My Tips After Hundreds Of Prompts

  • Give it a reference file. "Match the style of [existing-tool].html" is the cheapest way to get consistent output.
  • Say "mobile-first". Otherwise you'll get a desktop-only layout.
  • Ask for localStorage. Any tool that holds data should save it. Tell the AI the exact key name.
  • Be explicit about "no external libraries". Otherwise it may reach for React or Tailwind out of habit.
  • Iterate in small steps. Build v1, test it, then ask for feature 2. Don't ask for everything at once.
You Still Have To Review Everything
AI is a power tool, not a pilot. It can write bad code, miss edge cases, and occasionally make things up. Read what it produces. Test the tool. Fix the mistakes. You'll learn more by reviewing than by typing every line yourself, but only if you actually review it.

Step 10: Custom Domain (Optional)

15 min

A custom domain like yourname.com is purely cosmetic — your site works fine at the free .github.io URL. But if you want one, it's quick.

Buy A Domain

  • Namecheap — ~$10/yr for .com, easy UI.
  • Porkbun — similar pricing, my personal favorite.
  • Skip GoDaddy. Their renewals are eye-watering.

Point It At GitHub

  1. In your domain registrar's DNS settings, add these records:
    DNS Records
    Type   Name  Value
A      @     185.199.108.153
A      @     185.199.109.153
A      @     185.199.110.153
A      @     185.199.111.153
CNAME  www   yourusername.github.io
  2. In your GitHub repo → Settings → Pages → Custom domain, type yourdomain.com and hit save.
  3. GitHub will automatically create a file called CNAME in your repo with your domain in it. Don't delete it.
  4. Wait 5–30 minutes for DNS to propagate. Then check the "Enforce HTTPS" box for free SSL.
That's It
Your site is now at yourdomain.com, encrypted, hosted free forever. Hard to beat that.

Step 11: Publishing Updates

Reference

Once you've installed Git and VS Code, the workflow for updating your site is four commands. Run them in Terminal (Mac) or Git Bash (Windows) from inside your repo folder.

Git Cheat Sheet
# clone your repo (one time)
git clone https://github.com/yourusername/yourusername.github.io.git
cd yourusername.github.io

# after editing files, publish them:
git add .
git commit -m "added a new tool"
git push

# get the latest changes (if editing from another machine)
git pull

After git push, wait about 60 seconds and GitHub Pages will rebuild automatically. Refresh your site. Done.

Plain English Translation
add = "stage these changes". commit = "save a snapshot with this message". push = "upload to GitHub". pull = "download what's on GitHub". That's 95% of what you'll ever need.
